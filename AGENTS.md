# AGENTS.md

## 目的

Study Canvasは、iPadの手書きキャンバスで受験勉強の計画を作成する個人用Webアプリです。最重要価値は「今日何をするか決められること」です。

## 作業ルール

- 作業開始前に最新のmainを基準にする。
- mainは正式版として扱い、機能追加は作業ブランチとPull Requestで行う。
- 一度に1機能を基本とし、必要な箇所だけ変更する。
- iPad Safariの横向きを最優先し、指でも操作できる大きさを維持する。
- iPad Safariで意図しないダブルタップ拡大を発生させない。
- ページ一覧のサムネイルは既存のベクターデータから生成し、画像を重複保存しない。
- 実装後は`npm test`と`npm run build`を実行する。
- 不具合があれば修正と再テストを最大5回繰り返す。
- コンフリクト記号、秘密情報、重大なコンソールエラーを残さない。
- コード変更と同じPull RequestでPROJECT_STATUS.md、TODO.md、DECISIONS.mdを必要に応じて更新する。
- 公開画面をユーザーが確認するまで「iPad Safariで動作確認済み」と記録しない。
- 日付別保存へ移行した後も、旧`study-canvas:drawing:v1`データは自動削除しない。

## 低リスク変更の自動化

次の条件をすべて満たす変更は低リスクとする。

- 保存データの読み書き、削除、保存形式を変更しない。
- 外部API、APIキー、有料サービス、ログイン、データベースを追加・変更しない。
- 依存パッケージ、GitHub権限、セキュリティ設定、公開方式を変更しない。
- 大規模な構造変更ではなく、影響範囲が限定されている。
- ユーザーデータを失う可能性がない。

例: 文言、スタイル、保存データに触れないUI、テスト、管理ファイル、影響範囲が限定された不具合修正。

低リスク変更は、以下をすべて満たした場合、ユーザーの追加承認なしでAIがmainへのマージ、GitHub Pagesへの公開、公開後確認まで進める。

- Pull Requestの自動テストとビルドが成功している。
- mainの最新変更を取り込み、マージコンフリクトがない。
- コンフリクト記号と秘密情報がない。
- 主要機能と既存機能に問題がない。
- 公開後にサイトが開き、重大なコンソールエラーがない。

公開後に問題が見つかった場合は最大5回の修正ループ内で直す。安全に直せない場合は完成扱いにせず、問題と次の対応を報告する。

## 高リスク変更

次のいずれかに該当する変更は、実装またはマージ前にユーザーの承認を得る。

- ユーザーデータの削除、保存処理、保存形式、移行処理の変更。
- 大規模なコード構造の変更。
- 外部API、APIキー、有料サービス、ログイン、外部データベースの追加。
- 依存パッケージの追加・大幅更新。
- セキュリティ設定、GitHub権限、ブランチ保護、公開方式の変更。
- 公開中のアプリに大きな影響を与える変更。
- 危険度を明確に判断できない変更。

高リスクの承認依頼では、コードではなく「変わること・危険性・戻し方」を初心者にも分かる日本語で説明する。

## 現在変更しないもの

- AIによる手書き文字認識
- タスクカード
- ログインと外部データベース
- AI家庭教師、自動計画、自動予定修正
